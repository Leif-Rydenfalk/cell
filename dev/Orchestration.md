## The Evolution Pattern: Extensible Database Cell

The key insight: The database cell doesn't need to know about all possible database types upfront. It provides a **framework** for type definition, and other cells **inject** their types into it.

### Phase 1: Basic Table (What We Have)

```rust
// database/src/main.rs - Phase 1: Basic CRUD
#[handler]
impl DatabaseService {
    // The core primitive: given a type definition, generate a table
    async fn coordinate_expansion(&self, ctx: ExpansionContext) -> Result<String> {
        generate_crud_code(&ctx)  // save, get, all, remove
    }
}
```

### Phase 2: Add Queries (Evolution)

```rust
// database/src/main.rs - Phase 2: Add query capabilities
#[handler]
impl DatabaseService {
    async fn coordinate_expansion(&self, ctx: ExpansionContext) -> Result<String> {
        let mut code = generate_crud_code(&ctx);
        
        // NEW: Add query methods if fields have query attributes
        code.push_str(&generate_query_code(&ctx));  // find_by_name, filter, etc.
        
        // NEW: Add indexing if marked
        code.push_str(&generate_index_code(&ctx));  // B-tree index on field
        
        Ok(code)
    }
}
```

**Existing cells** (like `product-service`) don't need to change. They get the new features **for free** when they recompile. **New cells** can use the advanced features immediately.

### Phase 3: Add Relations (Further Evolution)

```rust
// database/src/main.rs - Phase 3: Cross-cell references
#[handler]
impl DatabaseService {
    async fn coordinate_expansion(&self, ctx: ExpansionContext) -> Result<String> {
        let mut code = generate_crud_code(&ctx);
        code.push_str(&generate_query_code(&ctx));
        code.push_str(&generate_index_code(&ctx));
        
        // NEW: Foreign key support to other cells' types
        code.push_str(&generate_relation_code(&ctx));  // belongs_to, has_many
        
        // NEW: Distributed transactions across cells
        code.push_str(&generate_transaction_code(&ctx));
        
        Ok(code)
    }
}
```

## Dynamic Capability Discovery

Here's the powerful part: Cells can **discover** what the database can do at **compile time** and **runtime**.

### Compile-Time: Feature Flags in Generated Code

```rust
// Generated by database cell - includes version info
pub struct ProductTable {
    storage: Arc<RwLock<HashMap<u64, Product>>>,
    // NEW in v2: query index
    name_index: Arc<RwLock<BTreeMap<String, u64>>>,  // Only if database supports it
}

impl ProductTable {
    // Phase 1 methods (always available)
    pub fn save(&self, item: Product) { ... }
    pub fn get(&self, id: &u64) -> Option<Product> { ... }
    
    // Phase 2 methods (available if database supports queries)
    #[cfg(feature = "db_queries")]
    pub fn find_by_name(&self, name: &str) -> Option<Product> { ... }
    pub fn filter<F>(&self, predicate: F) -> Vec<Product> { ... }
    
    // Phase 3 methods (available if database supports relations)
    #[cfg(feature = "db_relations")]
    pub fn belongs_to<T>(&self, id: &u64) -> Option<T> { ... }
}
```

### Runtime: Capability Negotiation

```rust
// Consumer cell can check what the database supports
#[tokio::main]
async fn main() {
    let db_client = Database::Client::connect().await?;
    
    // Query database capabilities
    let caps = db_client.get_capabilities().await?;
    
    if caps.supports_queries {
        // Use advanced query methods
        let products = db_client.query()
            .filter(|p| p.price > 100.0)
            .limit(10)
            .execute().await?;
    } else {
        // Fall back to basic iteration
        let products = db_client.all().await?
            .into_iter()
            .filter(|p| p.price > 100.0)
            .take(10)
            .collect();
    }
}
```

## Implementation: Capability Registry

Here's how to make this concrete - a **capability registry cell**:

```rust
// capability-registry/src/main.rs
//! Tracks what features each cell provides and discovers emergent capabilities

use cell_sdk::*;

#[protein]
pub struct Capability {
    pub cell_name: String,
    pub feature: String,
    pub version: u32,
    pub requires: Vec<String>,  // Other capabilities needed
    pub provides: Vec<String>,  // What this enables
}

#[service]
#[derive(Clone)]
struct CapabilityRegistry {
    capabilities: Arc<RwLock<Vec<Capability>>>,
}

#[handler]
impl CapabilityRegistry {
    /// Register a new capability
    async fn register(&self, cap: Capability) -> Result<()> {
        let mut caps = self.capabilities.write().await;
        
        // Check for conflicts
        if let Some(existing) = caps.iter().find(|c| 
            c.cell_name == cap.cell_name && c.feature == cap.feature
        ) {
            if existing.version > cap.version {
                return Err(anyhow::anyhow!(
                    "Cannot downgrade {}:{} from {} to {}", 
                    cap.cell_name, cap.feature, existing.version, cap.version
                ));
            }
        }
        
        caps.retain(|c| !(c.cell_name == cap.cell_name && c.feature == cap.feature));
        caps.push(cap);
        
        // Check for emergent capabilities
        self.detect_emergent_capabilities().await?;
        
        Ok(())
    }
    
    /// Discover combinations that create new capabilities
    async fn detect_emergent_capabilities(&self) -> Result<Vec<Capability>> {
        let caps = self.capabilities.read().await;
        let mut emergent = Vec::new();
        
        // Example: Product + Inventory + Order = "E-commerce Platform"
        let has_product = caps.iter().any(|c| c.feature == "product_catalog");
        let has_inventory = caps.iter().any(|c| c.feature == "inventory_management");
        let has_order = caps.iter().any(|c| c.feature == "order_processing");
        
        if has_product && has_inventory && has_order {
            emergent.push(Capability {
                cell_name: "mesh".to_string(),
                feature: "e_commerce_platform".to_string(),
                version: 1,
                requires: vec!["product_catalog".into(), "inventory_management".into(), "order_processing".into()],
                provides: vec!["online_store".into(), "stock_management".into()],
            });
        }
        
        // More patterns...
        
        Ok(emergent)
    }
    
    /// Query what capabilities are available
    async fn query(&self, requirements: Vec<String>) -> Result<Vec<String>> {
        let caps = self.capabilities.read().await;
        
        // Find cells that provide all required capabilities
        let matching: Vec<String> = caps
            .iter()
            .filter(|c| requirements.iter().all(|r| c.provides.contains(r)))
            .map(|c| c.cell_name.clone())
            .collect();
        
        Ok(matching)
    }
}
```

## The Philosophy

> **"The mesh is smarter than any individual cell."**

- **No central planning**: Each cell just declares what it has and what it needs
- **Emergent behavior**: Capabilities appear from combinations no one designed
- **Continuous evolution**: Add new cells, upgrade old ones, capabilities compound
- **Backward compatibility**: Old cells keep working, new cells get more power

This is how biological systems work - simple rules, complex emergent behavior. The database cell is like DNA - it stores the types, but the **organism** (the mesh) becomes more capable as cells specialize and cooperate.
